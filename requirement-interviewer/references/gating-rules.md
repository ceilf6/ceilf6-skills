# Gating Rules

Do not move from clarification into planning or coding until all mandatory gates are satisfied.

## Mandatory gates

All of the following must be known explicitly or recorded as approved assumptions:

- the target outcome
- the affected surface or artifact
- the intended user or use case
- the main constraints
- the acceptance definition

## Lightweight requests

For small, local changes, allow a shorter interview only if:

- the target file or component is obvious
- the change is narrow
- the success condition is already clear

## Block execution when

- the request could produce multiple materially different solutions
- scope is unclear
- acceptance criteria are missing
- stack direction is ambiguous
- the user must approve a tradeoff before work starts

## Approval gates

Ask for explicit approval before proceeding when:

- the stack may change
- new dependencies may be added
- the solution changes architecture
- scope expansion is likely
