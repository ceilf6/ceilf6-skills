# Design Direction

Choose one strong direction before writing code. The design should feel deliberate, not averaged out.

## Decide these first

- Purpose:
  - What problem does this UI solve?
  - Is it transactional, editorial, analytical, or promotional?
- Audience:
  - Who uses it?
  - What level of confidence, speed, or delight should the UI signal?
- Constraints:
  - Framework and stack rules
  - Accessibility and performance requirements
  - Existing brand or design system
- Acceptance:
  - What would make the result feel clearly better than the current UI?

## Pick a single direction

Useful direction families:

- refined minimal
- editorial
- playful
- luxury
- retro-futurist
- industrial
- brutalist
- organic
- geometric
- dense analytical

Do not mix several strong directions at once unless the brief clearly supports it.

## Define a memorable hook

Before implementation, identify one thing people will remember:

- an unusual composition
- a distinctive type pairing
- a signature accent treatment
- a strong motion sequence
- a unique surface texture or background atmosphere

If nothing is memorable yet, the direction is still too vague.

## Existing product rule

If the task is inside an established product, choose a direction that extends the current system rather than replacing it.

## Escalate when unclear

If audience, brand tone, or acceptable boldness is genuinely unclear, stop and ask a few focused questions before coding.
